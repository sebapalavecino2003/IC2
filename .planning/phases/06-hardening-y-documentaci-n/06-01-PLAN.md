    **Add `buildStatusJson()` method** — follows the same pattern as `buildTelemetryJson()` but builds a status payload:

    ```cpp
    void MqttManager::buildStatusJson(char* buffer, size_t buffer_size)
    {
        // Get current system state as string
        const char* state_str = "UNKNOWN";
        if (m_state_mach) {
            state_str = m_state_mach->getStateName();
        }

        // Get uptime from esp_timer
        uint32_t uptime_ms = (uint32_t)(esp_timer_get_time() / 1000);

        // Get free heap
        uint32_t free_heap = (uint32_t)esp_get_free_heap_size();

        // Build minimal status payload (extensible)
        snprintf(buffer, buffer_size,
            "{\"device_id\":\"%s\",\"status\":\"online\",\"uptime_ms\":%lu,"
            "\"free_heap\":%lu,\"system_state\":\"%s\","
            "\"last_reset_reason\":%d}",
            DEVICE_ID, (unsigned long)uptime_ms,
            (unsigned long)free_heap, state_str,
            (int)esp_reset_reason());
    }
    ```

    Required new includes at top of file (mqtt_manager.cpp):

    - `#include "esp_system.h"` (for esp_reset_reason)

    **Add `publishStatus()` method** — follows the same pattern as `publishReading()` but:
    - Topic: `buildTopic("status", topic, sizeof(topic))`
    - QoS: 1 (not 0)
    - Payload from `buildStatusJson()`
    - Uses `esp_mqtt_client_publish(m_client, topic, payload, 0, 1, 0)` (QoS 1)

    ```cpp
    void MqttManager::publishStatus()
    {
        char topic[64];
        char payload[256];

        buildTopic("status", topic, sizeof(topic));
        buildStatusJson(payload, sizeof(payload));

        if (m_connected && m_client) {
            int ret = esp_mqtt_client_publish(m_client, topic, payload, 0, 1, 0);
            if (ret < 0) {
                m_error_handler->reportError("MQTT", "Status publish failed");
            }
        }
        // Do NOT buffer status messages — they are time-sensitive, stale status is worse than no status
    }
    ```

    **Note on state transition publishing (D-13):** The mqttTask loop reads sensor readings and can detect state changes via m_state_mach. The 60s heartbeat is the primary mechanism; state transitions are covered implicitly because the status payload includes `system_state`, so the next 60s publish will reflect any recent transition. This is a reasonable trade-off per the agent's discretion.

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| ESP32 firmware → MQTT broker | Untrusted network transport for MQTT messages |
| ESP32 internal: FreeRTOS scheduler → Task execution | Intra-device boundary where a stuck task blocks sensor reads |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-06-01 | Denial of Service | FreeRTOS tasks (any of 7) | mitigate | Task WDT at 10s timeout (esp_task_wdt_init + esp_task_wdt_add per task). CPU hang caught by Interrupt WDT (Kconfig). Both trigger esp_restart() |
| T-06-02 | Tampering | Last Will MQTT message | accept | MQTT credentials protect publish; low-value status data, no PII, local network only |
| T-06-03 | Information Disclosure | Status MQTT payload | accept | No PII or secrets in status payload (free_heap, uptime, state_name, reset_reason). Local network only |
| T-06-SC | Tampering | esp_task_wdt.h / esp_int_wdt.h (ESP-IDF components) | mitigate | Official ESP-IDF components (no third-party packages). Slopcheck N/A for IDF components |
</threat_model>
